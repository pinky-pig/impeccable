## 什么时候使用

`/impeccable` 是整个体系的基础层。它会先把“怎么做设计”这件事交给你的 AI：设计原则、反模式、字体、色彩、布局判断，全都从这里开始。其他命令几乎都建立在它之上。

当你希望直接进入自由设计模式，并一次性加载整套设计判断时，就直接用 `/impeccable`。它还有两个常用子模式：

### /impeccable craft {#craft}

完整的「先 shape，再 build」流程。它会先在内部运行 `/shape`，围绕目标、用户、限制和氛围做结构化发现，然后直接进入实现与可视化迭代。适合全新功能，你想先想清楚再开工，但又不想自己拆每一步。

### /impeccable teach {#teach}

一次性的项目初始化。它会用简短访谈了解你的品牌、受众与审美方向，然后在项目根目录写入 `.impeccable.md`。之后所有设计相关命令都会自动读取这份上下文。

### /impeccable extract {#extract}

从现有代码里抽取可复用组件、design tokens 与模式，沉淀进设计系统。适合产品功能积累到一定阶段后，重复模式已经足够明显的时候再做。抽得太早，只会制造抽象层。

## 它是怎么工作的

大多数 AI 生成界面的问题，不是“代码不对”，而是“判断太泛”。默认字体、默认渐变、默认卡片网格、默认暗色霓虹，这些都是没有明确方向时最容易滑进去的模板。

`/impeccable` 会先建立一个清晰的设计立场，再决定怎么写代码。它内置了 **Context Gathering Protocol**，如果还没有项目上下文，就会先要求你运行 `/impeccable teach`。这是故意的：没有上下文的设计，最后大概率都会长成模板化界面。

## 试一下

先在一个新项目里运行一次：

```
/impeccable teach
```

回答完问题后，技能会写出一份 `.impeccable.md`，把品牌、受众与审美方向固定下来。

然后再让它做具体界面：

```
/impeccable build me a pricing page for a developer tool
```

理想结果应该是：它会坚定选择一种审美方向，主动避开默认字体与默认 AI 配色，并做出真正有判断的页面。

## 常见误区

- **跳过 `/impeccable teach`。** 没有 `.impeccable.md` 时，技能只能在执行中途重新追问上下文，效率会低很多。
- **把它当成样式规范。** 它不是冷冰冰的 linter，而是带立场的设计搭档。你当然可以反驳它，但最好给出明确理由。
- **希望它直接修现有页面。** `/impeccable` 更适合创建。如果是已有页面的修正与提升，优先考虑 `/polish`、`/distill` 或 `/critique`。
