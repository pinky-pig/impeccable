## 什么时候使用

`/harden` 适合“界面终于遇到真实世界”的那一天。真实数据从来不整齐：60 个字符的人名、德语标题、十亿级数字、空列表、500 错误、离线模式、RTL 文本……只能在完美数据下正常工作的设计，还不能算生产可用。

它特别适合上线前、准备进入新市场之前，或者每当 bug 报告以“某个用户名字特别长”开头的时候。

## 它是怎么工作的

这个技能会从 5 个维度给界面补强：

1. **文本与数据极值**：超长文本、超短文本、特殊字符、emoji、RTL、超大数字、超长列表、零数据状态。
2. **错误场景**：网络失败、API 4xx/5xx、校验错误、权限错误、限流、并发操作。
3. **国际化**：长翻译文本、RTL 语言、日期与数字格式、货币符号、不同字符集。
4. **引导与空状态**：首次使用、空状态设计、渐进式披露、功能发现路径。
5. **设备与上下文**：触控目标、离线场景、慢网速、低功耗模式。

每个维度它都会先找出失败模式，再给出具体修正：文本溢出处理、真正可用的空状态、信息充分的错误界面、i18n 安全的布局、正确的复数与格式化、必要的回退策略。

## 试一下

先从一个页面和一个问题开始：

```
/harden the user profile page for long names
```

常见输出可能包括：

- `.user-name` 加上 `text-overflow: ellipsis`，并为完整值提供 tooltip
- `.bio` 从固定高度改为 `max-height` + “展开更多”
- 为没有个人简介的用户补上空状态
- 为异步头像加载补 skeleton
- 测试长度从 1、20、60 到 200 字符的不同情况

最好按页面逐步执行，而不是一次把整个产品都扔进去。第一次改动通常最大，后续会越来越收敛。

## 常见误区

- **等 bug 出现再补。** Harden 更适合预防。某一类问题如果已经重复出现两次，就该整页跑一遍。
- **把错误与空状态当作收尾。** 很多补强工作，真正花时间的就是这些 UI，不只是写一个 `catch`。
- **因为暂时只做英文就跳过 i18n。** 事实上，能安全支持 i18n 的布局，本身就是更健壮的布局。
